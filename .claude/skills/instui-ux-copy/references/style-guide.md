## About this guide

This style guide is for the product team at Instructure. It includes writing guidelines that help us create a clear and consistent user experience.

Questions or additions? Visit the #ux-writing-help Slack channel and complete the "Get style guide help" workflow.

## Voice and tone

### Voice (always the same)

Our "voice" is the verbal demonstration of our brand character. It shows up everywhere we do — from our site to our ads to our keynote speeches — and it aligns with our company values:

- Ownership
- Relationships
- Equality
- Openness
- Simplicity

Our brand voice is always:

**Clear**
We prefer simple, straightforward language to industry jargon. We make dense information and challenging topics easier to understand and tough decisions easier to make.

**Confident**
This doesn't mean we're obnoxious egomaniacs — it means we're not wishy-washy. We have swagger. We love what we do, we know the market, we make great edtech, and we sound like it.

**Conversational**
Human beings want to hear from other human beings. We use contractions where people use them (because when you are reading something that does not use contractions, it is like reading the writing of a robot). We acknowledge shared human experiences. We make jokes and quips. We empathize and commiserate.

---

### Tone (changes with context)

If "voice" is our personality or character, then "tone" is our mood. While we always speak in the same voice, we often adjust our tone so that it's appropriate to the user's needs and emotional state.

We consider these factors to decide the right tone:
- Stage of user journey
- User emotions
- Space for content

| Tone | Description | Avoid |
|------|-------------|-------|
| Knowledgeable | Conveys expertise | Condescending know-it-all |
| Encouraging | Inspires, offers judgment-free support, reminds teachers how powerful they are | Patronizing |
| Sincere | Shows we hear and care about educators' challenges and goals | Schmaltzy, manipulative |
| Playful | Pop culture references, wordplay, friendly winks | Snarky, childish, alienating |
| Direct | Gets to the point quickly | Abrupt, no introduction or transition |

## Writing for everyone

### Plain language

We write in a way that helps people take the next best step quickly. Using plain language helps a wider audience understand and use a product.

**Why use plain language?**
- Accessibility: Required under WCAG. Supports comprehension, addresses secondary language and cognitive challenges, and helps assistive technology parse information correctly.
- SEO: Plain language uses the words and phrases people use when they search for content.
- Reading behavior: People scan online content. They're less likely to drop off when content is easily found and understood.
- Business benefits: Plain language can increase customer compliance, reduce complaints and errors, improve sales, and reduce costs.

**Recommended limits:**
- Word length: No more than 3 syllables
- Sentence length: No more than 25 words
- Paragraph length: No more than 150 words
- Reading level: 8th grade or lower (use the Hemingway App to check)

**Plain language principles:**

| Principle | Guidance |
|-----------|----------|
| Keep it short | Concise writing helps the audience quickly understand and use content |
| Keep it simple | Use common, everyday words |
| Be conversational | Write in an approachable way. Read it out loud to check. |
| Use contractions | Makes copy more conversational and human-sounding |
| Use active voice | Subject of a sentence performs the action |
| Avoid hidden verbs | Don't turn verbs into nouns |
| Use present tense | Future and conditional tenses increase cognitive load |
| Use second-person | Address the reader as "you" and "yours" |
| Avoid jargon | Don't use specialized words unless the audience definitely knows them |
| Use headers | Add structure and increase accessibility |
| Use lists | Make content easy to scan and remember |

**Preferred simple words:**

| Do | Don't |
|----|-------|
| Help | Assistance |
| Get | Access |
| Get rid of | Eliminate |
| Help | Facilitate |
| Ask | Request |
| Need | Require |
| To | In order to |
| If | In the event that |
| About | In regard to |
| Pay | Make a payment |

**Avoid hidden verbs:**

| Do | Don't |
|----|-------|
| Except for | With the exception of |
| Applies to | Is applicable to |
| Review | Leave a review |

### Inclusive language

We believe all of our customers deserve respect and hospitality. We use inclusive language to help promote dignity, enhance communication, and combat discrimination and bias.

**General guidance:**
- Only refer to someone's identity when it's relevant
- Whenever possible, ask people how they prefer to be referred to
- If the person's pronoun is unknown, use gender-neutral pronouns "they/them/their"
- Default to person-first language for people with disabilities (unless they specify otherwise)
- Default to identity-first language when describing race (unless they specify otherwise)
- Avoid language, metaphors, and idioms that exclude, offend, or assume a certain class, culture, ability, or age
- Avoid culturally insensitive slang or appropriating cultural terminology

**Disabilities:**

| Do | Don't |
|----|-------|
| Person with a disability | Disabled person |
| Person without a disability | Able-bodied person |
| Has a disability | Suffers from a disability |
| Disabled | Differently-abled |
| Overlooked spot | Blind spot |
| Show/view more | See more |
| Unbelievable | Crazy/insane/dumb |
| Feature off/deactivated | Feature disabled |
| In the menu, before, next, [specific language] | On the left, above, below, here |

**Race and culture:**

| Do | Don't |
|----|-------|
| Black person | Person who is black |
| Underrepresented groups | Minorities |
| Legacied | Grandfathered |
| It's been a while | Long time no see |
| No | No can do |
| Authority/expert | Guru |
| Meeting | Powwow |
| Let's be transparent | Let's open the kimono |

**Gender:**

| Do | Don't |
|----|-------|
| LGBTQ+ | Homosexual |
| Sexual orientation | Sexual preference |
| Pronouns | Preferred pronouns |
| Women and men | Female and male |
| Hey everyone | Hey guys |
| Everyone | Ladies and gentlemen |
| Businessperson | Businessman |
| Staffed/managed | Manned |
| Workforce | Manpower |
| Parental leave | Maternity/paternity leave |

## Grammar and mechanics

### Acronyms
- Spell out unusual acronyms on first use with the acronym in parentheses. Use only the acronym thereafter.
- Exception: Skip spelling out if users definitely already know it.
- Time zones: Use the three-letter abbreviation with a space before it (e.g., "8 am CST")

### Active vs. passive voice
- **Default: active voice** — subject performs the action
- **Use passive voice when:** avoiding blame on the user or system, or putting users in the spotlight

| Do | Don't |
|----|-------|
| We'll find the right person to help you. | The right person can be found to help you. |
| The password requirements weren't met. | You didn't meet the password requirements. |
| Your password was updated. | We updated your password. |

### Addresses
Abbreviate addresses as much as possible.

### Alt text
Write alt text for: images/icons/buttons/graphics, chart data, audio/video descriptions, form labels.

Rules:
- Don't write alt text for purely decorative content
- Provide a short, clear description of the information or function
- Don't include "image," "icon," or "picture" (screen readers announce this automatically)

### Ampersands (&)
- OK in headings and CTAs when there's a hard character limit
- Never in full sentences or after commas

### Asterisks (*)
Use to mark required fields (e.g., `Enter password*`)

### Capitalization
We default to **sentence case** — only capitalize the first word in a phrase and any proper nouns.

Note: Historically both title case and sentence case were used. Sentence case is now the standard going forward.

### Colons
- Use to introduce bulleted lists, numbered lists, and inline lists
- Don't use to introduce radio buttons or checkboxes
- Don't capitalize the first word after a colon — unless the colon is followed by a complete sentence

### Contractions
- Use contractions to sound conversational
- Avoid conditional contractions: could've, should've, would've

### Dashes

| Type | Use | Spacing |
|------|-----|---------|
| Hyphen (-) | Connect two+ modifier words that should be understood together | No spaces |
| En dash (–) | Ranges | No spaces |
| Em dash (—) | Break in sentence, asides, open ranges | Spaces in UI; no spaces in longer-form content |

Don't hyphenate "very" or words ending in -ly.

### Ellipses
Only use for truncation.

### Emojis
- Rarely — only in celebratory moments
- Never more than once per page
- Never to replace a word

### Exclamation marks
- Only in moments of celebration
- Never more than once per page

### First vs. third person
- **Default in product:** first-person plural ("We're working on a solution.")
- **Third person OK for:** brand copy, legal copy, help center content, references to Instructure in relation to customers/products

### Headers
- Use contextual, descriptive headers
- H1: page title only — one per page
- H2: second-level headings
- H3: third-level, use sparingly
- Don't skip heading levels
- Use sentence case

### Links
- Link text must be meaningful and describe the destination
- Don't link punctuation at the end of inline CTAs

### Lists
- Use a colon to introduce
- Limit to 6 items or fewer
- Order items logically
- Keep items similar in length
- Don't end items with commas
- Only end items with periods if the item is a complete sentence
- Use numbered lists when order matters

| Do | Don't |
|----|-------|
| Get started: 1. Create an account 2. Confirm your email 3. Adjust your preferences | Get started by visiting the website, creating an account, and adjusting your preferences. |

### Oxford comma
Always use.
- ✅ Collaborate, innovate, and succeed.
- ❌ Collaborate, innovate and succeed.

### Numbers

**Spell out:**
- In a sentence
- To de-emphasize a figure
- Large numbers: combine numeral and word (e.g., "4 billion")

**Use numerals:**
- For numbers 10 and above
- Time, tables, equations, ranges, dates, percentages
- To emphasize a figure

### Dates
- Spell out whenever possible: `February 7, 2025`
- Use shorthand only with restricted character limits or when asking users to manually input a date
- Don't use two digits for single-digit dates: `February 7` not `February 07`
- Numerical format (U.S.): `MM/DD/YYYY`
- Numerical format (non-U.S.): `YYYY/MM/DD`
- Ranges within one year: year only at end — `Feb 21–Feb 22, 2025`
- Ranges across years: year in both dates

### Phone numbers
Format: `xxx-xxx-xxxx` — link whenever possible.

### Time
- Lowercase `am` or `pm` with a space before: `8 am`
- Use `:00` when the time is on-the-hour and listed alongside other times
- Use "to" for ranges when space allows, en dash otherwise
- Omit first am/pm if range is within the same part of day: `8–10 am`
- Time zone: three-letter abbreviation with space: `8 am CST`

### Parentheses
- Use for context, counts, or abbreviations
- Period goes outside closing parenthesis (unless entire sentence is inside)

### Periods
Use only at end of complete sentences. Don't use in:
- Buttons
- Headers, labels, titles
- Alerts (unless full sentences)
- Month, day, or time abbreviations
- Short, direct, standalone phrases

### Semicolons
Never use. Use an em dash instead or rewrite the sentence.

### Text formatting
- **Bold:** To emphasize a point or reference navigation
- *Italics:* Not used in product (OK in marketing/longform)
- Underline: Links only
- ~~Strikethrough~~: Never — not accessible
- Vertical bars `|`: Separate metadata (e.g., `Math | CCSS | 6th Grade`)

## Common content

### Smails
- Total length: under 150 words
- Paragraphs: one topic, no more than 4 lines (~50 words)
- Use subheaders and bulleted lists for complex info

| Element | Guidance |
|---------|----------|
| Subject line | Under 10 words or 40 characters. No punctuation. Lead with deadline or required action. |
| Pre-header | Under 10 words or 40 characters. Complete sentence with punctuation. Reinforces or continues subject line. |
| Header | Optional. Highlights main message. One line ideally. |
| Body | Concise. One topic per paragraph. Max 3 lines per paragraph. |
| CTA | Always include at least one. Subject line + header + CTA alone should convey the main message. |
| Footer | Optional secondary CTA. Always include: business address, unsubscribe link, app download links. |

### Legal disclaimers
- Serious tone
- Max 4 lines on mobile
- Include a link to a resource for more information
- Work with legal on what's required in the UI

### Push and SMS notifications
- Under 90 characters
- Time-sensitive, actionable messages only
- Send push first; only send SMS if notifications are off

| Element | Guidance |
|---------|----------|
| Header | One line. Main message. Occasional emoji for celebratory. |
| Body | Max 3 lines. Indicate where user will navigate on click-through. |

### Calls-to-action (buttons and links)

**Rules:**
- Sentence case (unless proper noun)
- Buttons: 4 words or fewer
- Links: 8 words or fewer
- Use buttons for primary actions; links for navigation
- Follow [action verb] + [noun] pattern (unless common pattern like "Done" or "Continue")

**The 4 S's (Nielsen Norman Group):**
- **Specific:** Start with strong, actionable verbs. Avoid "click."
- **Sincere:** Accurately describe where the user will go.
- **Substantial:** Make copy understandable without surrounding context.
- **Succinct:** Cut articles and unnecessary adjectives.

| Do | Don't |
|----|-------|
| View policy details | Learn more |
| Start free trial | Try it now! |
| Set reminder | Set a reminder |

### Errors and alerts

**Principles:**
- Explain what happened and what to do next
- Avoid generic language
- Never use exclamation points
- Use passive voice to avoid blaming the user
- Don't make jokes

| Element | Guidance |
|---------|----------|
| Header | One line, max 10 words. Communicates what went wrong. Header + CTA alone should resolve the issue. |
| Body | Complete sentences, max 2 lines. Additional context on reason for error. No technical jargon (e.g., "Error 404"). |
| CTA | 4 words or fewer. No punctuation. Direct user to resolve if possible. |

| Do | Don't |
|----|-------|
| The payment failed. Double-check your information and try again. | AUTH ERROR: We weren't able to process your payment. |
| Incorrect password | You entered the wrong password. |
