## Copy and content issues

### "E-mail address" — outdated hyphenated spelling

**What was written:** `E-mail address`
**What it should be:** `Email address`
**Which guideline:** Plain language — use current, common spellings. "Email" (no hyphen) is the standard.
**Root cause hypothesis:** The original kitchen-sink reference code predated the style guide integration. No hook or ESLint rule catches spelling conventions in JSX string props.
**Suggested fix:** Add "E-mail" to the tone-check skill's explicit violation list. Consider adding it to the PostToolUse copy-detection hook pattern.

### "Login" used as a verb CTA

**What was written:** `<Button>Login</Button>`, `<Link>Login</Link>`
**What it should be:** `Log in` (two words as a verb)
**Which guideline:** CTAs follow `[action verb] + [noun]` pattern. "Login" is a noun; the verb form is "log in."
**Root cause hypothesis:** The reference code used "Login" throughout and the write-microcopy/tone-check skills weren't yet present when the reference was authored. Common mistake — "login" as a single word is widespread in codebases even when wrong.
**Suggested fix:** Add "Login" as a verb to the tone-check violation list with the explicit rule: use "Log in" (two words) for the action, "login" (one word) only as a noun (e.g. "login page").

### Exclamation marks in non-celebratory strings

**What was written:** `"Register now!"`, `"You can switch themes in the prototype here!"` (×3)
**What it should be:** `"Register"`, `"Select a theme to preview how components look."`, etc.
**Which guideline:** Style guide — exclamation marks only in moments of celebration, never more than once per page.
**Root cause hypothesis:** Placeholder copy written without a style guide reference. The three repeated instances of the themes string suggest copy-paste from the first card without review.
**Suggested fix:** The PostToolUse copy-detection hook could grep for `!` inside JSX string children and emit an advisory. Add to the tone-check violation list.

**What it should be:** `"Button — colors"`, `"Icon button — shapes and sizes"`, `"Text input"`, etc.

### "screenReaderLabel" describing color/state rather than action

**What was written:** `screenReaderLabel="Primary"`, `screenReaderLabel="Secondary"`, `screenReaderLabel="AI Primary"`, etc. on `IconButton` components in the kitchen-sink showcase.
**What it should be:** Labels that describe the action the button performs, not its visual variant (e.g. `"Star item"`, `"Open notifications"`, `"Open AI assistant"`).
**Which guideline:** Style guide alt text rules — describe function, not appearance. Screen reader labels must make the button's purpose clear without visual context.
**Root cause hypothesis:** The kitchen-sink is a component showcase, and the `screenReaderLabel` values were written to label the visual variant being demonstrated rather than a real action. This is understandable in a demo context but teaches a bad pattern.
**Suggested fix:** Add a note to the tone-check skill's post-review step: flag `screenReaderLabel` values that match color names, state names, or size names (Primary, Secondary, Small, Large, Disabled, etc.) as potential accessibility gaps. Add a comment in the kitchen-sink source acknowledging these are demo-only labels.

### Placeholder text in delivered reference output

**What was written:** `"Page title"` (h1), `"Primary action"` / `"Secondary action"` (buttons), `"Tab item"` (×4), `"Content area"` (h2), `"Short card description"`, `"Level 1"` / `"Current Level"` (breadcrumbs), lorem ipsum body text — all in `CanvasPage.tsx` at the time of the first tone-check run.
**What it should be:** Real, contextually appropriate copy (resolved: "Assignments", "Create assignment", "Overview" / "Details", "Upcoming assignments", "Due in the next seven days", "Courses" / "Introduction to Biology", real body text).
**Which guideline:** Tone-check skill — placeholder scan step. Style guide — no filler text in shipped UI.
**Root cause hypothesis:** The `CanvasPage.tsx` template was built as a structural reference, not a copy reference. Placeholders were intentionally deferred. The tone-check skill did not exist at authoring time, so there was no gate to catch them. After the skill was added to the harness, the first tone-check run correctly flagged all of them — the system worked as intended once the skill was present.
**Suggested fix:** The PostToolUse copy-detection hook should extend its pattern matching to include common placeholder strings ("Page title", "Tab item", "Content area", "Lorem ipsum", "Primary action"). This would have caught these at write time rather than requiring a manual skill invocation.

### "Add a brief description here — one or two lines works best." left as visible UI text

**What was written:** `<Text>Add a brief description here — one or two lines works best.</Text>`
**What it should be:** Either real copy or a code comment — not rendered UI text.
**Which guideline:** Placeholder scan — instructional meta-copy rendered as visible text is a placeholder.
**Root cause hypothesis:** The agent improved on the original placeholder ("This is a page description. If your page requires describing…") by writing something more helpful, but still left it as a rendered `<Text>` node rather than converting it to a JSX comment.
**Suggested fix:** When the tone-check skill flags a placeholder, the suggested remediation should explicitly offer two options: replace with real copy, or convert to a `{/* */}` comment. The skill currently only offers the first option.

## General design issues

**What happened:** The second "Switch themes" card in `kitchen-sink/index.tsx` had a heading of "Switch themes" but its content was a tag selector, number input, range input, and confirm/back buttons — nothing to do with theme switching.
**Evidence:** Identified during the first tone-check run. The description also read "You can switch themes in the prototype here!" which was copy-pasted from the card above it.

## Patterns and recommendations

- **Extend the PostToolUse hook to catch common placeholders at write time:** The hook currently detects sentence-like strings in JSX for the tone-check advisory. Extending it to also grep for known placeholder patterns ("Lorem ipsum", "Page title", "Tab item", "Primary action", "Content area") would catch these at the moment they're written rather than requiring a manual skill run later.

- **Add a verb-vs-noun CTA check to the tone-check skill:** "Login" vs "Log in", "Signup" vs "Sign up" are a consistent class of error. The skill should call these out explicitly with a rule and examples, not just the general "action verb + noun" principle.

- **Add a screenReaderLabel quality check to tone-check:** Labels that match color names or state names (Primary, Secondary, Disabled, Readonly) are almost always wrong in production context. The skill should flag these and prompt for action-describing alternatives.

- **Update the tone-check placeholder remediation to offer comment conversion:** When a placeholder is in a template file (any file in `src/templates/`), the default suggestion should be to convert the `<Text>` to a `{/* */}` comment, not to write real copy — since template files exist to be copied and specialized, not shipped as-is.

- **Consider a `screenReaderLabel` ESLint rule:** An ESLint rule that warns when `screenReaderLabel` matches a set of known non-descriptive patterns (color names, state names, generic nouns) would catch accessibility labeling errors at write time. This is a gap the existing three custom rules don't cover.

- **The two-layer auto-invocation design (AGENTS.md + hook) worked:** The tone-check skill was invoked proactively by the agent on the reference prototypes without the user needing to name the skill — consistent with the AGENTS.md behavioral instruction. The hook pattern is sound; the remaining gap is the breadth of placeholder patterns it detects.
